5.114. simplified-ui-catalog

5.114.1. Issues

5.114.1.1. Fragment Backstack Used For Navigation History

1
The current UI uses Android Fragments to construct the catalog interface. The previous version of the UI used Android Activities. In order to have the fewest surprises in terms of implementation, the old Activity-based UI was more or less translated directly to Fragment form. That is, each loaded feed is an instance of a Fragment, and the Android Fragment backstack is used to store the history of feeds through which the user has navigated. Unfortunately, the Android Fragment backstack is fraught with issues and cannot be adequately controlled to give a pleasant user experience.
2
Additionally, our Fragment instances are stateful, and this state can be invalidated. For example, if the user navigates to a feed in account A, and then navigates to a feed in account B, this will result in a fragment F being placed on the backstack that has a reference to information in account A. If the user then navigates to the settings screen and deletes account A, the Fragment F now contains references to the deleted account A. Any attempt to move F back to the foreground and interact with it in any meaningful way will almost certainly crash the application. The current unpleasant workaround we apply is to simply clear the entire backstack whenever an account is deleted. This is why the user is unceremoniously thrown back to the catalog screen whenever they delete an account.
3
The proper way to implement this would be to store the navigation history as a navigable data structure in a ViewModel, and use one stateless Fragment instance instead of multiple instances.
4
Issue class: DESIGN. Severity: Critical.

5.114.1.2. Navigation Feeds Unsupported

1
The catalog UI does not currently support navigation feeds. This simply makes some OPDS feeds outside of the Library Registry unusable in the application.
2
Issue class: DESIGN. Severity: Major.

5.114.1.3. Poor Scrolling Performance

1
Infinitely-scrolling feeds still do not have acceptable scrolling performance. The application will typically stutter when scrolling the first 20-30 elements of feed, after which scrolling performance improves.
2
Issue class: BUG. Severity: Minor.

5.114.1.4. Infinite Feed Navigation Is Imprecise

1
The model used to store both the in-memory representation of infinitely scrolling feeds, and the user's navigation history in the application is inadequate. This means that it is essentially impossible to navigate back to a specific point in an infinitely scrolling feed after having switched to a different feed.
2
Issue class: DESIGN. Severity: Minor.

5.114.1.5. Age Gate Should Not Be A Dialog

1
Because of a miscommunication between various parties, a new COPPA age gate was added to comply with Google's Play Store requirements. The new age gate was added as a dialog, when what should have happened was that the existing full-screen age gate should simply have been updated with new text and controls.
2
Issue class: DESIGN. Severity: Minor.

5.114.1.6. No Acquisition Controls

1
The application does not allow users to pick specific acquisitions when borrowing or downloading books. A written specification exists that describes the selection process, but it is unclear if all of the Library Simplified applications follow it.
2
Issue class: DESIGN. Severity: Minor.

5.114.1.7. Book Problem Reports Unimplemented

1
The application used to contain a way for users to report specific problems with books. Unfortunately, the code for this was broken and the UI was eventually removed. To the present day, it has not been reimplemented.
2
Issue class: DESIGN. Severity: Minor.

5.114.1.8. Search Is Insufficient

1
The catalog search dialog is slightly clumsy and each search query results in a new feed page being loaded each time. For reasons of aesthetics, this could/should be replaced with something that loads search results in-place.
2
Issue class: DESIGN. Severity: Minor.

5.114.1.9. Feeds Requiring Authentication Are Obtrusive

1
A somewhat ugly loop exists in the user interface. If the user navigates to a feed that requires authentication in order to view it, the user is forcefully redirected to a login screen. If they try to navigate back without logging in, they are forced back to the login screen. This can make it next to impossible to switch accounts.
2
Issue class: DESIGN. Severity: Major.

5.114.1.10. Books View Hides Books From Other Accounts

1
The "Books" tab (the tab showing books that the user currently has loaned or downloaded) only shows books from the most recent account. The historical reason for this is that support for multiple accounts was added in very early versions of the application in a very haphazard way. In early versions of the application, the application was almost entirely restarted to point at a new account each time the user switched accounts, and there was no way to access data from accounts other than the current account. The application was rewritten to properly support multiple accounts many years ago, but the "Books" tab still only has a user interface that displays books from one account rather than from all accounts. This has, effectively, turned what was a serious implementation bug into a conscious UI design decision.
2
Issue class: DESIGN. Severity: Major.

5.114.2. Caches

5.114.2.1. Fragment Instances

1
Fragments store downloaded feeds in a ViewModel. The ViewModel does not survive the fragment being removed from the fragment backstack, and is not strictly a cache in the traditional sense.

5.114.3. Module Link

1
The module can be found at simplified-ui-catalog.
